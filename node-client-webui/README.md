# 🚀 Node Client Web UI Project

This project provides a containerized environment using **Docker Compose** to launch the **Node Client Web UI** with automatic **self-signed TLS certificate generation**.

---

## 📂 Project Structure

```
docker-compose/
├── docker-compose.yml
├── .env.example
├── data/       # Persistent application data
├── openssl/    # Certificates and Keys folder
└── README.md
```

---

## 🔐 Environment Variables and Authentication Configuration

The Node Client Web UI credentials are configured using environment variables loaded from a .env file.

**Configure Credentials**

Copy the example environment file and edit it:
```bash
cp .env.example .env
```

Edit .env and set the following values:
```
EXTERNAL_IP=IP_OF_MY_MACHINE
WEBUI_USERNAME=MyUsername
WEBUI_PASSWORD=MyPassword
```
 - WEBUI_USERNAME – Username used to log in to the Web UI
 - WEBUI_PASSWORD – Password used to log in to the Web UI
 - EXTERNAL_IP - IP address or DNS name used to access the Web UI.

### 📦 Certificates
The Node Client Web UI supports automatically generated self-signed TLS certificates and private keys, as well as user provided certificates.


#### 🔐 Self-Signed Certificates (Default)
Self signed TLS certificates and private keys are automatically generated by **Node Client Web UI** and stored in the `openssl` directory if no existing certificates are detected. These certificates are reused on subsequent starts unless the files are deleted.

⚠️ Important note: These certificates are not trusted by browsers by default. You must accept them manually when accessing the UI.

#### 🔑 User-Provided Certificates
If you want to provide your own TLS certificates, you must generate them manually and place them in the `openssl` directory **before starting the container**.

📄 Required files:
 - TLS certificate: `./openssl/tls.crt`
 - TLS private key: `./openssl/tls.key`

When valid certificate files are detected in `./openssl`, automatic self-signed generation is skipped, and the provided certificates are used instead. The container logs will display: `TLS cert/key already exist; reusing.`

⚠️ Important notes:
 - Certificates must exist before running `docker compose up`
 - Do not remove the openssl directory
 - File names and formats must match what the container expects: `./openssl/tls.crt` and `./openssl/tls.key`.
 - If files are invalid or unreadable, startup may fail.

💡 Tip:
To switch back to self-signed certificates, stop the stack, remove the contents of `openssl` directory, and start the stack again.

### 🧹 Clean Up

When you are finished, you may remove containers, networks, and volumes.

- **Stop services**:
  ```bash
  docker compose stop
  [+] Stopping 1/1
   ✔ Container node-client-webui  Stopped                                                                                                         10.2s
  cluster-admin@nategateway-vm:~/docker-compose-node-wizard-webui$
  ```

- **Remove containers, networks and volumes**:
  ```bash
  docker compose down -v

  [+] Running 1/0
   ✔ Container node-client-webui  Removed                                                                                                          0.1s
  ```
  ⚠️ This will delete all persistent data.

- You may also delete the openssl and data directories if you want to redeploy with fresh certificates and state.


### 💡 Tips

- 🐧 **Using Podman instead of Docker**
  You can run this project with [Podman](https://podman.io/) as a drop‑in replacement for Docker.
  Simply replace `docker` with `podman` in all commands — the syntax is the same.


- 🔑 **Running Docker without `sudo`**
  By default, you may need `sudo` to access the Docker Engine.
  To allow your user to run Docker commands without `sudo`, add yourself to the `docker` group:

  ```bash
  # Create the docker group if it doesn't exist
  sudo groupadd docker

  # Add your user to the docker group
  sudo usermod -aG docker $USER

  # Apply the new group membership (log out and back in)
  ```


- 🔑 **Running Podman without `sudo`**
  By default, you may need `sudo` to run Podman command because the user socket is not running.
  ```bash
  # Enable the user socket
  systemctl --user enable --now podman.socket
  systemctl --user status podman.socket
  ```

## 🛠️ Troubleshooting

- **Login fails with "Network Error".**
  The default deployment uses **self-signed certificates**.
  Both the backend and frontend certificate must be trusted by the browser.

  ✅ **Fix:**
  Open the backend URL once and accept the certificate:
  ```
  https://EXTERNAL_IP:23051
  ```
  Then reload the frontend and try logging in again.
